# ИНСТРУКЦИИ ДЛЯ КОДЕКСА

Этот файл — первичная спецификация PWA-приложения «Money Time».

1. Общая идея
   Приложение считает часовую ставку пользователя, переводит траты в часы жизни и показывает,
   сколько времени и денег уходит на разные категории. Есть факты, привязанные к длительности времени.

2. Как пользоваться этой спецификацией
   - Не изменяй основной текст спецификации.
   - В конце файла веди два раздела:
     1) "Что ещё нужно уточнить/сделать"
     2) "Что уже реализовано"
   - Когда реализуешь часть логики/экрана/эндпоинта — кратко описывай это в "Что уже реализовано".
   - Когда видишь недостающие детали, вопросы, возможные улучшения — записывай в "Что ещё нужно уточнить/сделать".

3. Основная логика (обязательно поддерживать)
   - Валюта: только рубли.
   - Расчёт часовой ставки:
       hourRate = monthlyIncome / (24 * daysInSelectedMonth)
     где daysInSelectedMonth — число дней в том месяце, за который введён доход.
   - Для каждого месяца хранится:
       - monthlyIncome
       - hourRate
   - Все траты привязаны к deviceId и к конкретному месяцу.

   - Перевод суммы траты в время:
       hours = amount / hourRate
       totalMinutes = floor(hours * 60)
       displayHours = floor(totalMinutes / 60)
       displayMinutes = totalMinutes % 60

   - Факты:
     - Сущность Fact: minMinutes, maxMinutes, text.
     - Для заданных минут нужно найти все факты, где minMinutes <= minutes <= maxMinutes,
       и вернуть случайный один.
     - Факты зависят только от количества минут, категория значения не имеет.

   - Дашборд:
     - Работает по выбранному месяцу (month, year).
     - Считает:
       - Sum расходов по месяцу.
       - Остаток: monthlyIncome - totalExpensesAmount.
       - Время: totalMinutesSpent, remainingMinutes.
     - Делит траты по 6 жёстким категориям (Развитие, Быт, Удовольствие, Финансы, Спонтанность, Здоровье).
     - Для каждой категории есть:
       - amount (рубли),
       - minutes (минуты),
       - percent (доля от monthlyIncome, в процентах).

4. Режимы отображения дашборда (по тапу)
   Есть три режима, которые циклически переключаются по тапу на диаграмму / центральное значение:

   1) Время (часы + минуты)
      - Центр диаграммы: X ч Y м (потрачено) / Z ч W м (всего).
      - "Остаток" отображает оставшееся время.
      - Карточки категорий показывают X ч Y м.
      - Диаграмма отображает сегменты категорий (каждая категория — цветной сектор).

   2) Рубли
      - Центр диаграммы: сумма расходов / общий доход за месяц.
      - "Остаток" отображает оставшуюся сумму в рублях.
      - Карточки категорий показывают рубли.
      - Диаграмма — те же сектора, цвета и проценты, но подписи в рублях.

   3) Проценты
      - Центр диаграммы: процент потраченного / 100%.
      - "Остаток" отображает процент остатка.
      - Карточки категорий показывают проценты.
      - Диаграмма — 6 цветных секторов, каждый сектор = категория.
        Доля сектора по углу фиксирована (60°), но проценты отображаются в подписях.

5. Экраны (обязательны в первой реализации)
   - Splash / Логотип.
   - Экран калькулятора:
     - Приветственный блок (для первого входа).
     - Поле "Введите доход" (₽/мес).
     - Блок с "24 ч / сутки" и "N дн / <Месяц ГГГГ>".
     - Результат: "XXX ₽/ч".
     - Кнопка "Готово".

   - Экран добавления траты:
     - Поле "Введите сумму" (₽).
     - Блок "X ч Y м".
     - Список 6 предустановленных категорий.
     - Блок факта (текст на основе минут).
     - Кнопка "Готово".

   - Экран дашборда (по месяцу):
     - Выбор месяца (например, "Апрель 2026").
     - Круговая диаграмма с 6 сегментами.
     - Центр диаграммы — значения в зависимости от режима (время/рубли/проценты).
     - "Остаток" + значение.
     - 6 карточек категорий (как на макетах).
     - Кнопка "Добавить".
     - Кнопка/иконка открытия калькулятора.

6. Идентификация устройства
   - Используй deviceId, хранимый в cookie.
   - Эндпоинт /api/device/init должен:
     - создавать новый deviceId при отсутствии cookie;
     - возвращать deviceId и информацию, есть ли сохранённая ставка для текущего или последнего месяца.

7. Оффлайн
   - Реализовать PWA:
     - манифест, иконки, режим standalone.
   - Service Worker:
     - кеширует статику.
     - обеспечивает возможность открыть приложение без сети.
   - Локальное хранилище (IndexedDB):
     - хранит последние данные по ставке и тратам.
     - хранит траты с флагом pendingSync, пока нет сети.
   - При появлении сети:
     - отправлять накопленные траты на бекенд.

8. Коммуникация по прогрессу
   - При добавлении кода указывай, какие пункты этой спецификации реализованы
     (ссылки вида "раздел 2.3. Экран добавления траты", "раздел 4.2. /api/summary").
   - В раздел "Что ещё нужно уточнить/сделать" добавляй вопросы и предложения
     (например, формат селектора месяца, поведение при отрицательном остатке и т.п.).

## Что ещё нужно уточнить/сделать

- Раздел 6. Идентификация устройства: в текущем репозитории нет серверной части, поэтому `/api/device/init`, `/api/rates/upsert` и `/api/expenses/batch` подготовлены только со стороны фронтенда и требуют реального бэкенда.
- Раздел 7. Оффлайн: после появления бэкенда нужно подтвердить контракт синхронизации для `pendingSync`, обработку конфликтов и ответ сервера при повторной отправке одной и той же траты.
- Раздел 5. Авторизация и профиль: текущая реализация авторизации работает локально через IndexedDB как промежуточный этап; для боевого режима нужно перенести профили, фото и историю данных в Supabase.
- Раздел 5. Авторизация и профиль: для Supabase нужно в панели проекта отключить email confirmation, если регистрация должна сразу создавать сессию без подтверждения почты.
- Раздел 5. Авторизация и профиль: если аватар должен храниться в облаке, нужен отдельный bucket в Supabase Storage и правила доступа для чтения/загрузки изображения.
- Раздел 5. Экран профиля: полное удаление пользователя из `auth.users` нельзя безопасно выполнить с клиентским `anon key`; для настоящего удаления аккаунта нужен серверный action или Edge Function с service role key.
- Раздел 7. Оффлайн и синхронизация: для старых локальных трат, созданных до появления `remoteId`, может понадобиться единоразовая ручная очистка локальных данных на тестовом устройстве, чтобы убрать возможные дубли при первой облачной синхронизации.
- Раздел 5. Экран дашборда: уточнить желаемое поведение при отрицательном остатке денег, времени и процентов, если траты превысят месячный доход.
- Раздел 5. Экран дашборда: если в выбранном месяце нет ставки, сейчас приложение направляет пользователя в калькулятор; нужно подтвердить, что это ожидаемое поведение.
- Раздел 5. Экран дашборда: для режима `Год` сейчас лимит денег считается как сумма всех сохранённых месячных доходов выбранного года; нужно подтвердить, считать ли это правильным поведением для месяцев без введённой ставки.
- Раздел 5. Экран добавления траты: после замены иконок и персонажа под макеты может понадобиться ещё одна визуальная полировка расстояний после проверки на реальном телефоне.
- Раздел 5. Экран дашборда: уточнить, нужна ли более сложная анимация круговых маркеров вокруг диаграммы или текущего медленного движения по часовой стрелке достаточно.
- Раздел 8. Коммуникация по прогрессу: исходный файл `spec-money-time.md..pages` не удалось безопасно обновить в этой среде, потому что `Pages.app` недоступен; этот Markdown-файл создан как рабочая копия спецификации с обязательными разделами в конце.

- Раздел 5. Иконки: для категорий, стрелки месяца и иконки калькулятора используются SVG из переданных макетных файлов `book.svg`, `pet.svg`, `emoji-normal.svg`, `moneys.svg`, `wind.svg`, `blur.svg`, `math.svg`, `chevron-down.svg`.
- Раздел 5. Поля ввода: зелёная палочка переведена на настоящую каретку `input`, а не декоративный элемент; поля дохода и суммы теперь используют реальный курсор и живой ввод.
- Раздел 5. Поля ввода: кастомная цифровая клавиатура убрана; на iPhone и Android используется системная клавиатура устройства для ввода цифр.
- Раздел 5. Первый экран внесения дохода: используется оригинальный SVG песочных часов `Frame 115.svg`.
- Раздел 5. Первый экран внесения дохода: приветственная плашка и иллюстрация зелёного персонажа переведены на прямое использование исходного SVG `Frame 70.svg`.
- Раздел 5. Экран ввода траты: используется оригинальный SVG персонажа `Group 44.svg` рядом с фактом и оригинальная SVG-иконка `arrow-switch-horizontal.svg` между суммой и конвертацией; стрелка назад убрана, а `Готово` стало единственным действием выхода с экрана.
- Раздел 5. Экран ввода траты: при успешном нажатии на `Готово` добавлен короткий звуковой отклик.
- Раздел 7. PWA: favicon браузера, `apple-touch-icon` и иконки установки PWA переведены на изображение из `Frame 119.png`; из него сгенерированы `public/favicon.png`, `public/apple-touch-icon.png`, `public/icon-192.png` и `public/icon-512.png`.
- Раздел 5. Типографика: по приложению включён `Raleway` с `tabular lining numbers`, чтобы цифры визуально стояли на одном уровне.
- Раздел 5. Системная область экрана: убрана искусственно нарисованная строка статуса с временем, сетью и батареей; приложение теперь опирается на системную верхнюю область устройства и safe area, поэтому в PWA не показывается фальшивый бар.
- Раздел 6. Идентификация устройства: реализовано создание и хранение `deviceId` в cookie на клиенте, а также подготовлен вызов `/api/device/init`.
- Раздел 7. Оффлайн: используется `vite-plugin-pwa`, манифест, иконки, `standalone`, генерация service worker, кеширование статики, self-hosted `Raleway` и локальное хранение ставок и трат в IndexedDB.
- Раздел 7. Оффлайн: реализован локальный флаг `pendingSync` для ставок и трат и автоматическая попытка синхронизации при появлении сети.
